# StefARR V0.8 acquisition status

V0.8 separates three facts that older request views could blur together:

1. whether a request was approved and submitted;
2. what the acquisition backend is doing now; and
3. whether the result was imported and is visible in the target library.

A queue item disappearing is not success. StefARR only reports an import after
the responsible backend confirms a file, and reports availability only after
the target library confirms it.

## Status and actions

The Requests page groups work into active transfers, queued work,
post-processing/import, paused work, and problems. Supported providers can show
queue position, bytes, remaining time, and episode/file counts. Providers that
do not expose reliable byte progress show a lifecycle phase instead of a made-up
percentage.

Authorized request managers can retry a failed acquisition and acknowledge a
warning. Acknowledgement changes only StefARR's operational display; it never
deletes, pauses, or cancels downstream work.

## Privacy and retention

Provider download IDs, release names, indexers, and filesystem paths are used
only for server-side correlation and are never returned by the public Hub API.
Open normalized issues remain visible until they are resolved or the request is
cancelled. Resolved issues are removed after seven days.

Users see only acquisition data for requests they are already permitted to
view. Request viewers and managers retain their existing shared visibility.

## Upgrade and rollback

V0.8 adds an acquisition-issue migration for SQLite and PostgreSQL. Back up the
configuration directory and database before upgrading. The supported path is a
direct upgrade from V0.7.0. Downgrade is backup/restore only because older
versions do not understand the new issue table.

The first publication is `v0.8.0-beta.1`. Operators should validate queued,
active, post-processing, failed, imported, and available examples before
promoting the stable `v0.8.0` release.
